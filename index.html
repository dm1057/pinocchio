Pinocchio - A git puppet.

A minimalist tool for Git based server provisioning.

Why?

Server configuration and software provisioning should be damn simple.

There are lots of tools for helping you install and configure your full software stack at your new and shiny cloud based slice. Just to name a few, we have Puppet, Chef, and the Capistrano based Sprinkle. All of them trying to solve the problem of automating software installation and configuration on many servers. Basically you write a recipe file containing instructions on how to install or configure something, some of these recipes require you to learn their respective mini/dsl language, some of them even require you to host those recipes on a service of their own. Sure these tools all have their own merit and can be useful when managing lots of servers, but for me, they were sometimes just like too much complex when I just wanted to provision a new slice to get more fun at coding a new experiment.

I just wanted to have some dead simple way of uploading a new configuration file or instructions for the server on how to install something. I wanted to deliver these just like I'm used with regular code, by using git. I wanted to push and see only the new pending instructions being executed on my remote slices. I wanted to have all of it versioned and under control. I wanted these changes to be incremental, something like using rails database migrations but for provisioning my servers.

So Pinocchio was born.

How it works

Pinocchio is just a simple git update hook. It works by determining which changes it needs to execute since the last successful update. It does it by leveraging on git to know which migrations need to be run, and only when all of these pending migrations have a successful exit status the git repo is updated.

Installation

To use Pinocchio on your remote server, all you need is for it to have git installed and the pinocchio's update hook configured at a bare repo.

In the following examples we assume the remote repo is hosted at my-new-slice with user ubuntu at the ~/pinocchio directory.

Automated setup on server

The pinocchio install command can automatically setup Pinocchio for you on your remote server. In essence it just checks that git is installed or installs it for you if it can and then creates a new git bare repo at the specified directory, copying pinocchio's update hook and making it executable.

$ git clone https://github.com/vic/pinocchio.git && cd pinocchio
$ bin/pinocchio install '~/pinocchio' ubuntu@my-new-slice
Manual setup on server
# install git unless you already have it on your remote server
$ ssh ubuntu@my-new-slice 'sudo apt-get install git'

# create a remote repo for pinocchio and install the update hook
$ cat hooks/pinocchio | ssh ubuntu@my-new-slice 'git init --bare ~/pinocchio && cat -> ~/pinocchio/hooks/update && chmod 755 ~/pinocchio/hooks/update'

Usage

Clone Pinocchio into a local directory on your computer, and edit your remotes accordingly, for our previous ubuntu example it would be:

$ git remote --set-url origin ubuntu@my-new-slice:~/pinocchio

Of course you can create as many remotes as you wish to provision with your repo.

Creating migration scripts

After cloning your local pinocchio, you can create your provisioning migrations under the migrate/ directory. 

README.md
migrate/
        201311051205-install-mysql.sh

Pinocchio migrations are language agnostic, they can be just shell scripts as well as python or ruby scripts or whatever your server can execute. The only requirement for them is to be executables and exit with zero status when successful.

I highly recommend to prefix the migration files with a timestamp as Pinocchio will try to run them in natural order. To aid with timestamps you can use the pinocchio generate command:

$ bin/pinocchio generate install-mysql.sh
  Generated migrate/201311051205-install-mysql.sh
Configuration files

Pinocchio provides a files/ directory where you can place any file that will be copied into your servers filesystem whenever it changes.

README.md
migrate/
        201311051215-install-nginx.sh
files/
        etc/
             nginx.conf

Files are copied only after the pending migrations have finished successfully. So for example, when the install-nginx migration finishes, the files/etc/nginx.conf file gets copied into /etc/nginx.conf.

Note that Pinocchio makes no attempt at creating the directory structure, it simply copies the files and expects the migrations to have already prepared everything else.

User permissions

Sometimes you'd want the migration scripts to be executed as a particular user or the copied files to have some particular owner.

Pinocchio supports specifying user/group permissions for both migration scripts and copied files. You specify them with the pinocchio user and pinocchio group commands.

README.md
migrate/
        201311051230-create-rails-user.sh
        201311051231-install-rails-rvm.sh
files/
      home/
           rails/
                 .bashrc

We want the migration for creating a user to be run as root, and the migration to install rvm to be run as the rails user. We also specify the owner for the bashrc file.

$ bin/pinocchio user root  migrate/201311051230-create-rails-user.sh
$ bin/pinocchio user rails migrate/201311051231-install-rails-rvm.sh
$ bin/pinocchio user rails files/home/rails/.bashrc
Updating your remote servers

Pinocchio lets you manage incremental changes on your server as a simple git repo. So, after you've written some migration scripts at migrate/ or placed some configuration files under files/ all you need if to simply make a git push and the pinocchio's update hook will take care of performing pending actions and updating files.

If all migrations run successfully, the code is updated and next time only new migrations will be executed.
